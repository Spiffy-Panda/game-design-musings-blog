a run is a seed

The Determinism Harness

A lost run is only worth banking if it's reproducible. Same seed, same inputs, byte-for-byte the same defeat — that's what makes an overrun shareable, replayable, and the whole roguelite loop hold together.

Determinism isn't a feature you switch on. It's three disciplines held at once — one seeded PRNG advanced only on ticks, a fixed iteration order, and integer money. Break any single one and shareable seeds, replay, and run-ghosts die quietly, months before anyone notices.

Every run of Minimalist Space Logistics is a seed. You own one hauler keeping an unwinnable war supplied; the run ends in an overrun and banks prestige. The pillar the whole design leans on is APR.5 — determinism: a run is its seed, and the same seed plus the same player inputs must reproduce that run exactly. Lose, and the loss is a shareable artifact — a code a friend can paste in to walk your exact collapse, a ghost the next run can race.

That promise is fragile in three specific places. This page makes each one tangible: a tiny shared toy sim — a handful of producers and consumers trading credits on a fixed tick — wired into three instruments. In each, you can break the discipline and watch reproducibility die in real time. The page itself is the proof: every number below is computed live by a mulberry32 PRNG and an integer fixed-point money type, and the harness re-runs itself to check.

This is not a hypothetical worry. A sibling MSL page in this same set — prestige-reseeding, where a banked loss re-seeds the map — literally hit a reseed-PRNG determinism bug: the post-loss world drew from a stream that wasn't reset cleanly, so "the same seed" stopped producing the same map. The harness below is the discipline that would have caught it.

D1Float vs fixed-point drift

The same stream of money operations, run two ways. On the left, balances accumulate as floating-point credits (the obvious choice — money is dollars-and-cents, cents are decimals, decimals are floats). On the right, the identical operations run as integer fixed-point: money is stored as whole milli-credits (1 cr = 1000), never as a float. Each "op" is one tick's worth of trades — a producer sells, a consumer pays, the hauler skims a margin — drawn from the shared seeded PRNG.

Watch the running difference between the two ledgers. With fixed-point it is exactly zero, forever. With floats it creeps — a fraction of a milli-credit per rounding, compounding — until the two sets of books, fed the very same transactions, no longer agree. The light goes red the first tick they diverge.

Float rounding is exactly how double-entry books silently stop balancing. Each op the float side rounds to representable binary; the error is tiny and unbiased-looking, but it never undoes itself. Crank the op count: the drift grows. The fixed-point side adds whole integers — there is no rounding, so the difference is provably, permanently zero.

D2Seed → world

One 32-bit number expands into an entire run. The seed initializes a single mulberry32 stream; the run draws from it in a fixed sequence to lay out producer nodes on a banded board, decide the enemy's fall order across the front, and set per-producer production jitter. The bottom tape shows the raw PRNG values as they're consumed — the highlighted cells are the draws that became this world's keyframes.

Change the seed and you get a different world. Re-enter the same seed and you get the byte-identical world back — same nodes, same fall order, same jitter. That's the share code: a handful of characters that fully regenerates the run.

D3Iteration order — the subtle killer

This is the one that hides best. Same seed, same money type, same everything — except the order in which the sim visits its agents to settle a shared, contended resource. Each tick, a pool of agents draws from a limited supply pot; whoever is processed first gets served first, so the final balances depend on iteration order.

Iterate in a fixed order (sorted by id) and two runs of the same seed land on byte-identical results. Iterate in a shuffled / hash-map order — the order you get for free from most language dictionaries, and which can differ run to run — and the same seed produces different results each time. Run it twice and compare.

★Rerun & compare — the permanent gate

The harness that makes all of this enforceable is one rule: run the toy sim twice under the current settings and demand the two state hashes match. This is the CI gate the real game runs on every commit — "reruns must match" — turned into a button. It folds the whole shared sim (seeded draws, fixed-point money, the iteration order you chose above in D3) into a single rolling FNV-1a hash of the final state, then does it again, and shows you the verdict.

Flip the toggles to break a discipline and watch the gate go red. Set them back and watch it go green. A green here is a continuous, machine-checkable proof that the run is a seed.

The aha, in one line: determinism isn't automatic — it's three disciplines held together (one seeded PRNG advanced only on ticks · fixed iteration order · integer/fixed-point money), and breaking any one silently kills shareable seeds, replay, and run-ghosts.

Toggle the four breakers one at a time. Each is a different way the same green gate goes red — and each maps to a discipline above: float → D1, shuffled → D3, unseeded/extra draw → D2's "advance only on ticks."

Risks, named

Determinism rot

The slow leak. An unseeded rand() slips into a code path; a hash-map iteration replaces a sorted one; a float accumulates where an integer should. Each is invisible in a single playthrough and only shows up as "my friend's seed gave me a different run" — by which point shareable seeds, replay, and run-ghosts have all quietly stopped working.

Mitigation the harness shows: the rerun-and-compare gate above, run on every commit. Two runs of the same seed must produce the same state hash; the moment a discipline rots, the gate goes red in CI, not in a player's bug report. It already caught a real reseed-PRNG bug in the prestige-reseeding sibling.

Cross-platform float nondeterminism

Even with a single seed and fixed order, floating-point math is not guaranteed identical across CPUs, compilers, or SIMD vs scalar paths — fused multiply-add, x87 80-bit intermediates, and aggressive optimization can each change the last bit. A run recorded on one machine then fails to reproduce on another, which is fatal for shared seeds specifically.

Mitigation the harness shows: keep money and any replay-critical state in integer/fixed-point (D1) — integer arithmetic is bit-exact everywhere. Floats, if used at all, stay in cosmetic-only paths that never feed the state hash.

Thread-timing-dependent state

Parallelize the agent update for speed and the order results land in can depend on which thread finished first — a non-deterministic iteration order wearing a performance costume. Same failure as D3, harder to spot because it's timing, not code, that varies.

Mitigation the harness shows: the replay-critical update is single-threaded and visits agents in a fixed sort order; parallelism is reserved for read-only work (rendering, the glass cockpit) that never writes back into the seeded state.

This is the foundation under every other MSL system — the emergent market, the scripted enemy, the money ledgers all assume it holds. Determinism is the violet thread that runs beneath the amber, rose, and cyan ones (APR.5). Get it right once, at the bottom, and a lost run becomes a thing you can hand to someone else.
