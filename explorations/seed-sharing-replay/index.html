Share a Lost Run

You lost. The war overran your hauler at t-something and banked some prestige. Here is the strange gift determinism hands you: that exact defeat is now a six-character code you can text to a friend — "this is the war that beat me; beat it yourself."

A run is a pure function of its seed. So a loss isn't gone — it's an artifact. The same code regenerates the same producers, the same enemy fall order, the same collapse, anywhere, on anyone's machine. The daily challenge, "beat my run," same-war leaderboards — that whole social layer falls out for free, with no servers in the sim. Just a seed.

In Minimalist Space Logistics you keep one hauler supplying an unwinnable war. Every run is a seed; every run ends in an overrun. Because the sim is deterministic (APR.5) — fixed ticks, one seeded PRNG, fixed iteration order, integer money — a finished run is fully reproducible. The same seed regenerates the same war byte for byte: the same map, the same fall order, the same events. That is the entire premise of this page: determinism is the substrate; sharing is the payoff.

The mechanism — why floats would rot it, why iteration order is the subtle killer, the rerun-and-compare CI gate — is the sibling page determinism-harness. This page doesn't re-derive it. It takes "the run is a seed" as given and shows what that buys the player: a shareable, replayable artifact and the social loop built on top of it (R.5 — the war is learnable and shareable).

1Seed → world: the run card

Type a seed, pick a chip, or roll a random one. From that single 32-bit number a mulberry32 stream is drawn in a fixed sequence to lay out a banded map (rear supply → forward line), decide which systems fall and in what order, place your producers and the forward depot, and write a one-line difficulty read. Nothing here is stored or fetched — it is computed from the seed, live, every time.

2Prove it's identical — replay

"Same seed = same war" is a claim. Here it is a button. We generate the run from the current seed, then generate it again from the same seed, and fold each world's keyframes (bands, node placements, fall order — the discrete, integer state, never a raw float) into an FNV-1a hash. If the sim is a pure function of the seed, the two hashes match exactly. That match is what a replay stands on: re-run the seed and you walk the identical collapse.

3Compare two seeds — feel the seed-space

One seed is a war; the space of seeds is every war. Generate two run cards side by side and the seed-space becomes tangible — different maps, different fall orders, different depots, different difficulty. Swap a seed and the whole card rewrites, deterministically. This is why a daily seed is interesting: out of billions of distinct wars, today everyone is handed the same one.

4The daily seed — everyone fights the same war

Now the payoff lands. Derive one seed from today's date and you have a daily challenge for free: every player, everywhere, is handed the identical war — same map, same fall order, same depot, same collapse pressure — and the only variable is how long you survive it. Post your survival time; compare against the same war, not a similar one. That is a same-war leaderboard with no servers in the sim — the seed is the shared state.

Risks, named

Seed stability across versions

A share code is only a promise if it means the same war on every build. The instant world-gen changes — a new producer type, a tweaked band layout, an extra draw inserted mid-sequence — the same seed expands into a different world, and every code anyone ever shared silently rots. "MSL-7F3A9C beat me" becomes a lie on the next patch. This is the quiet killer of seed-sharing games: the seed didn't change, the function did.

Mitigation: version the seed/world-gen format, not just the game — stamp each code with the gen-version (e.g. MSL2-7F3A9C) and keep old generators resolvable so historical codes still reproduce their original war, or explicitly mark them retired. Treat world-gen as a stable serialized format with migrations, the way you would a save file — never an implementation detail free to drift.

Determinism — the whole premise

Every claim on this page rests on one assumption: the sim is a pure function of the seed (and inputs). One unseeded rand() in a code path that touches state, one hash-map iteration order, one float where an integer belongs, and shared seeds diverge — your friend pastes your code and gets a different war, the daily leaderboard compares times across non-identical worlds, replays desync. The feature isn't "sharing"; the feature is "sharing only because the world is reproducible," and that reproducibility is fragile.

Mitigation: this is precisely what the sibling determinism-harness defends — one seeded PRNG advanced only on ticks, a fixed iteration order, integer money, and a rerun-and-compare gate in CI that fails the build the moment two runs of a seed disagree. The amber breaker in panel 2 is a miniature of that failure: one unseeded draw and the prove-identical hash splits.

Seed-sharing is the violet thread's dividend: get determinism right once, at the bottom (APR.5), and a lost run stops being a dead end. It becomes a thing you can hand to someone — learnable, replayable, beatable (R.5) — and the daily challenge, the "beat my run" dare, the same-war board all arrive without a single server simulating the game.
